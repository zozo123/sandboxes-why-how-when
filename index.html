The Sandbox
Shift

Docker made code portable. Sandboxes make code safe to run — because the author is no longer a human you trust. It's a model, mid-reasoning, writing code faster than anyone can review it.

The bet: this is a bigger inflection than containers were. Generation got cheap. Safe execution is the new bottleneck.

The shift, in one table

Two revolutions, one assumption flipped

Container era · ~2013

Trusted human writes the code

  • Problem solved — "works on my machine"
  • Goal — portability + reproducibility
  • Lifetime — long-lived, named, pets
  • Trust model — code is reviewed & vouched for
→

Sandbox era · now

A model writes the code

  • Problem solved — "run this untrusted thing safely"
  • Goal — isolation, instant, disposable
  • Lifetime — ephemeral, by the thousands, cattle
  • Trust model — guilty until proven safe

The isolation tech mostly already existed. What changed is who writes the code and how much of it there is. That's enough to move sandboxing from an edge case to the default substrate of agentic computing.

Act ① — Why

Five reasons the box exists

1

Untrusted author

The code came from a model. It can carry a prompt-injected payload, a hallucinated rm -rf /, or a typo'd package name that resolves to malware. You cannot eyeball every rollout.

2

Blast radius

Contain what it can reach: the filesystem, network egress, environment secrets, and — if isolation is weak — the host kernel itself. The sandbox is the wall, not a hope.

3

Reproducibility

Every run starts from identical, clean state. Without it, an eval's reward signal is just noise — you can't tell a real improvement from leftover state in /tmp.

4

Parallelism at scale

RL and evals need thousands of disposable environments at once. That's only affordable when a box is cheap to spawn, cheap to kill, and dense to pack.

5

State control

Snapshot, fork, and restore lets an agent branch its exploration without re-paying setup cost — try five fixes from the same checkpoint, keep the one that passes.

“If a human didn't read it, a sandbox should run it.”

Act ② — When

The lifecycle, and the trust boundary

Sandboxes show up at every stage where AI-authored code meets a real machine:

  1. execFast ephemeral runcode interpreter, ad-hoc AI-gen Python, data analysis
  2. devAgent coding loopsa branch / worktree per task, throwaway on fail
  3. ciTest & builda reproducible harness, no "works on my machine"
  4. evalEvals & RLreasoning sandboxes that run harnesses
  5. prodProduction toolsagent code execution serving real users

The phrase to remember

A reasoning sandbox that runs a harness

The eval / RL loop is the purest case. Each rollout is a fresh box:

// one rollout = one disposable sandbox
spawn  fresh sandbox        # clean, identical state
setup  the task             # clone repo @ ref, stage files
run    the agent's code     # the untrusted part
score  reward = harness()   # programmatic, no human grading
destroy everything          # ×1000, in parallel

The decision that actually matters

Inside the VPC, or outside it?

Two axes: how much you trust the code, and how sensitive the data it can reach.

Untrusted code →
No sensitive data ↓
Run it locally A sandbox is mostly overhead. Process limits are plenty.
Outside / public ephemeral Untrusted code, zero access to crown jewels. Pure compute, lowest blast radius. The default for arbitrary AI-gen code.
Private data / internal ↓
Inside VPC, light isolation Trusted code that needs real data. Standard internal tenancy.
Inside VPC, hard isolation The hard case: untrusted code that also needs private data or internal services. Isolation now contains something with real network reach. Compliance + data-residency stakes are highest here.

Act ③ — How

The isolation ladder

Weak and fast at the bottom, strong and heavy at the top. Pick the lowest rung that holds your threat model. (The decider above highlights one for you.)

  1. subprocessSame kernel, same user. A timeout and a ulimit. Trusted code only.
  2. namespaces · cgroups · seccompnsjail, bubblewrap. Cheap kernel-level fencing. Good for semi-trusted code.
  3. containerrunc / containerd. Shared kernel — one kernel CVE is an escape.
  4. gVisorA user-space kernel intercepts syscalls. Container ergonomics, smaller attack surface.
  5. microVM · FirecrackerReal VM boundary, boots in ~100 ms, thousands per host. The agent-sandbox sweet spot.
  6. full VM · air-gappedMaximum isolation, maximum cost. For the genuinely hostile.

The whole game in one line

isolation strength ↔ startup latency ↔ density / cost

microVMs win for agents because they break the old rule that "VM-grade isolation means slow and expensive." ~100 ms boots make per-run disposability economically real.

Operating patterns (regardless of tier)

  • Ephemeral by default — destroy after every run.
  • Egress deny-by-default; allow-list only what's needed.
  • Scope secrets — forward one token, not the whole environment.
  • Hard resource caps — CPU, memory, and wall-clock time.
  • Snapshot / fork to skip repeated setup cost.
  • Exfil results over a narrow channel (e.g. a stdout sentinel), not a shared mount.

Objections, answered

"But isn't this just…?"

…just containers?

Containers share the host kernel — one kernel CVE and you're out. Fine for code you trust, thin for unreviewed model output. That gap is exactly why gVisor and microVMs exist.

…just a VM with extra steps?

In isolation terms, yes — and that's the point. The novelty isn't the boundary, it's booting it in ~100 ms and packing thousands per host. That economics is what makes per-rollout disposability possible at all.

…hype? we've run untrusted code for decades.

True. cgi-bin, CI runners, online judges — none of it is new. What changed is volume and authorship: code is now model-authored, unreviewed, and generated faster than humans can vet. Isolation moves from special-case to default.

…overkill for my script?

Often, yes. If you wrote it and it touches nothing sensitive, a sandbox is overhead — the decider above will tell you to just run it. The point isn't "always sandbox." It's "match the box to the threat."

A converging landscape

Where the tooling is going

Not a recommendation — a map. Two broad camps, both trending toward fast, VM-grade, disposable isolation.

Self-hosted primitives

Docker · containerd · gVisor · Firecracker · Kata · nsjail · bubblewrap

You own the kernel boundary and the bill. Maximum control, maximum ops.

Managed sandbox platforms

E2B · Modal · Daytona · and a growing field

An API spawns the box for you. Less control over the substrate, far less to run.
